# x86 架构上的锁实现

> C++ 锁在 x86 上为什么这么简单？

---

## 1. x86 内存模型：TSO（Total Store Order）

x86 有相对较强的内存模型，很多重排被禁止：

```
┌─────────────────────────────────────────────────────────────┐
│  x86 禁止的重排：                                            │
│                                                             │
│  ❌ Store-Store 不会重排                                     │
│  ❌ Load-Load 不会重排                                       │
│  ❌ Load-Store 不会重排（对同一地址）                         │
│                                                             │
│  x86 允许的重排：                                            │
│                                                             │
│  ✅ Store-Load 可能重排（store buffer）                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

---

## 2. 锁在 x86 上的实现

### mutex::lock()

```cpp
void mutex::lock() {
    // x86 实现
    while (xchg(&flag, 1) == 1) {  // xchg 隐含 lock 前缀
        // 等待
    }
}
```

```asm
; 汇编
lock xchg [flag], eax
```

**`lock` 前缀做了什么**：

```
┌─────────────────────────────────────────────────────────────┐
│  lock xchg 的三个作用：                                      │
│                                                             │
│  1. 原子性：读-改-写不可分割                                 │
│     └─ 硬件保证这个操作期间，其他CPU不能修改同一缓存行       │
│                                                             │
│  2. 缓存一致性：获取独占权                                   │
│     └─ MESI协议确保其他CPU的副本失效                        │
│                                                             │
│  3. 内存屏障：全序屏障                                       │
│     └─ 前后的读写都不能越过这条指令                         │
│                                                             │
│  这三个作用是一条指令同时完成的                              │
└─────────────────────────────────────────────────────────────┘
```

### mutex::unlock()

```cpp
void mutex::unlock() {
    // x86 实现
    flag = 0;  // 简单的 store
}
```

```asm
; 汇编
mov [flag], 0
```

**为什么不需要 `lock` 前缀？**

```
┌─────────────────────────────────────────────────────────────┐
│  x86 TSO 保证：                                              │
│                                                             │
│  1. Store-Store 不会重排                                     │
│     └─ 临界区内的所有写入一定在 flag=0 之前完成              │
│                                                             │
│  2. MESI 保证可见性                                          │
│     └─ flag=0 这个写入会让其他CPU看到新的缓存行              │
│     └─ 其他CPU获取锁时，会触发MESI同步                      │
│                                                             │
│  所以 unlock() 在 x86 上就是一个普通的 store                 │
└─────────────────────────────────────────────────────────────┘
```

## 3. 对比：ARM 架构

```
┌─────────────────────────────────────────────────────────────┐
│  ARM（弱内存模型）需要显式屏障：                              │
│                                                             │
│  lock():                                                    │
│    ldaxr w0, [flag]    ; Load-Acquire                       │
│    ; ... 检查和循环 ...                                      │
│                                                             │
│  unlock():                                                  │
│    stlr w1, [flag]     ; Store-Release                      │
│    dmb ish              ; 显式内存屏障                      │
│                                                             │
│  x86 不需要这些，因为 TSO 本身就很强                         │
└─────────────────────────────────────────────────────────────┘
```

---

## 4. 理解总结

| 说法 | 是否正确 | 补充说明 |
|------|----------|----------|
| x86 只用原子指令 | ✅ 大部分正确 | `lock` 前缀同时提供原子性+屏障+MESI |
| 内存屏障用不上 | ⚠️ 部分正确 | `lock` 前缀本身就是全序屏障；x86 大多数情况不需要额外屏障，但 `mfence` 指令在特殊场景下仍需要 |
| CPU 可见性由 MESI 搞定 | ✅ 正确 | MESI 保证缓存一致性，其他CPU一定能看到修改 |
| x86 默认不乱序执行 | ⚠️ 不完全准确 | x86 有 Store-Load 重排（store buffer），但 `lock` 前缀阻止了这种重排 |

---

## 5. 关键点

```
┌─────────────────────────────────────────────────────────────┐
│  x86 上锁的实现本质：                                        │
│                                                             │
│  lock():  lock xchg  ← 一条指令搞定原子性+屏障+缓存一致性   │
│  unlock(): mov       ← 普通store，TSO保证顺序               │
│                                                             │
│  不需要：                                                   │
│  - 额外的 mfence 指令                                       │
│  - 显式的缓存刷新                                           │
│  - 编译器屏障（除非有特殊需求）                              │
│                                                             │
│  这是 x86 的优势：强内存模型让并发编程更简单                 │
└─────────────────────────────────────────────────────────────┘
```

---

## 6. 编译器优化仍然需要处理

```cpp
// 编译器可能重排的场景
data = 42;
flag.store(1, std::memory_order_release);

// 如果不指定 memory_order，编译器可能优化为：
flag = 1;      // 编译器把 flag 提前
data = 42;     // 编译器把 data 延后
```

```
┌─────────────────────────────────────────────────────────────┐
│  即使 x86 硬件不乱序，编译器也可能乱序                      │
│                                                             │
│  需要：                                                     │
│  - std::memory_order 指定语义                               │
│  - 编译器插入适当的屏障/约束                                 │
│                                                             │
│  硬件保证：x86 不乱序                                       │
│  软件保证：编译器不乱序                                      │
└─────────────────────────────────────────────────────────────┘
```

---

## 7. 总结

```
你的理解基本正确，x86 上锁的实现确实比 ARM 简单得多：

  x86:  lock xchg + 普通 store + MESI 自动搞定
  ARM:  ldaxr/stlr + 显式 dmb + MESI

但需要注意：
  1. lock 前缀不仅是原子指令，还提供内存屏障
  2. 编译器优化仍然需要正确的 memory_order
  3. MESI 保证可见性，但不保证顺序（顺序由 TSO 保证）
```
