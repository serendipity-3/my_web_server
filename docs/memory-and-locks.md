## 1. 数据存在哪里？

程序中的变量不是只存在一个地方。CPU 访问数据时，会经过多层存储。理解这些层次，是理解线程同步的基础。

### 三层存储结构

```
┌─────────────────────────────────────────────────────────┐
│                      主内存 (RAM)                        │
│                                                         │
│   int counter = 0;  ← 数据的"真相"存储在这里              │
│                                                         │
└───────────────────────┬─────────────────────────────────┘
                        │
          ┌─────────────┴─────────────┐
          │                           │
          ▼                           ▼
┌─────────────────────┐     ┌─────────────────────┐
│   CPU 0 的缓存      │     │   CPU 1 的缓存      │
│                     │     │                     │
│  L1: counter = 0    │     │  L1: counter = 0    │
│  L2: counter = 0    │     │  L2: counter = 0    │
│  L3: (共享)         │     │  L3: (共享)         │
└──────────┬──────────┘     └──────────┬──────────┘
           │                           │
           ▼                           ▼
┌─────────────────────┐     ┌─────────────────────┐
│   CPU 0 的寄存器    │     │   CPU 1 的寄存器    │
│                     │     │                     │
│  EAX: 0             │     │  EAX: 0             │
└─────────────────────┘     └─────────────────────┘
```

### 各层特点

| 层级 | 位置 | 访问速度 | 容量 | 共享性 |
|------|------|----------|------|--------|
| **寄存器** | CPU 内部 | 最快 (~1ns) | 几十个字节 | 每个 CPU 独立 |
| **L1 缓存** | CPU 内部 | 很快 (~4ns) | 32-64KB | 每个 CPU 独立 |
| **L2 缓存** | CPU 内部 | 快 (~10ns) | 256KB-1MB | 每个 CPU 独立 |
| **L3 缓存** | CPU 外部 | 较快 (~40ns) | 几MB-几十MB | 多个 CPU 共享 |
| **主内存** | 主板 | 慢 (~100ns) | 几GB-几十GB | 所有 CPU 共享 |

### 关键理解

**每个 CPU 都有自己的缓存副本。** 当 CPU 0 修改 `counter` 时，它可能只修改了自己的缓存，CPU 1 的缓存并不知道这个变化。

```
时间线：

T0: 所有缓存一致
    主内存: counter=0
    CPU0缓存: counter=0
    CPU1缓存: counter=0

T1: CPU0 修改 counter++
    主内存: counter=0  ← 还没更新
    CPU0缓存: counter=1  ← 只有这里变了
    CPU1缓存: counter=0  ← 不知道变化

T2: CPU1 读取 counter
    得到 0（过期数据！）
```

## 2. 没有锁时会发生什么？

### 场景设定

两个线程同时执行 `counter++`：

```cpp
#include <iostream>
#include <thread>

int counter = 0;  // 共享变量

void increment() {
    for (int i = 0; i < 100000; ++i) {
        counter++;  // 看似简单，实际很复杂
    }
}

int main() {
    std::thread t1(increment);
    std::thread t2(increment);
    
    t1.join();
    t2.join();
    
    std::cout << "counter = " << counter << std::endl;
    // 期望 200000，实际可能更小
    return 0;
}
```

### counter++ 的真实过程

`counter++` 不是一条指令，而是三条：

```
1. LOAD   - 从内存读取 counter 到寄存器
2. ADD    - 寄存器值 +1
3. STORE  - 将寄存器值写回内存
```

### 数据竞争的详细过程

```
初始状态：counter = 0

┌─────────────────────────────────────────────────────────────┐
│                        主内存                                │
│                    counter = 0                               │
└─────────────────────────────────────────────────────────────┘

时间 ──────────────────────────────────────────────────────────►

CPU 0 (线程1)                    CPU 1 (线程2)
────────────────                 ────────────────

T1: LOAD counter
    寄存器 EAX = 0
    │
    ▼
T2: ADD EAX, 1
    寄存器 EAX = 1
    │
    ▼
                                 T3: LOAD counter
                                     寄存器 EAX = 0  ← 读到旧值！
                                     │
                                     ▼
                                 T4: ADD EAX, 1
                                     寄存器 EAX = 1
                                     │
                                     ▼
T5: STORE EAX → counter
    主内存 counter = 1
    │
    ▼
                                 T6: STORE EAX → counter
                                     主内存 counter = 1  ← 覆盖了！
                                     │
                                     ▼

最终结果：counter = 1（丢失了一次增加）
```

### 为什么会出现数据竞争？

```
┌─────────────────────────────────────────────────────────────┐
│                        主内存                                │
│                    counter = 0                               │
└─────────────────────────────────────────────────────────────┘
         ▲                                    ▲
         │ LOAD                               │ LOAD
         │ (得到 0)                           │ (得到 0)
         │                                    │
┌────────┴────────┐                  ┌────────┴────────┐
│     CPU 0       │                  │     CPU 1       │
│                 │                  │                 │
│  EAX = 0        │                  │  EAX = 0        │
│      │          │                  │      │          │
│      ▼          │                  │      ▼          │
│  EAX = 0+1 = 1  │                  │  EAX = 0+1 = 1  │
│      │          │                  │      │          │
└──────┼──────────┘                  └──────┼──────────┘
       │                                    │
       │ STORE (写入 1)                     │ STORE (写入 1)
       ▼                                    ▼
┌─────────────────────────────────────────────────────────────┐
│                        主内存                                │
│                    counter = 1  ← 应该是 2！                  │
└─────────────────────────────────────────────────────────────┘
```

**核心问题**：两个 CPU 各自读取了 `counter=0`，各自加 1，各自写回 1。第二次写入覆盖了第一次。

## 3. mutex 的作用

`mutex`（互斥锁）确保同一时间只有一个线程能进入临界区。

### 基本用法

```cpp
#include <iostream>
#include <thread>
#include <mutex>

int counter = 0;
std::mutex mtx;  // 互斥锁

void increment() {
    for (int i = 0; i < 100000; ++i) {
        mtx.lock();      // 获取锁
        counter++;       // 临界区
        mtx.unlock();    // 释放锁
    }
}

int main() {
    std::thread t1(increment);
    std::thread t2(increment);
    
    t1.join();
    t2.join();
    
    std::cout << "counter = " << counter << std::endl;  // 总是 200000
    return 0;
}
```

### mutex 的数据流动

#### 阶段 1：获取锁（acquire）

```
┌─────────────────────────────────────────────────────────────┐
│                        主内存                                │
│                                                             │
│   counter = 0          mutex = UNLOCKED                     │
│       │                     │                               │
└───────┼─────────────────────┼───────────────────────────────┘
        │                     │
        │                     │ lock() 操作
        │                     │ (原子地将 mutex 改为 LOCKED)
        │                     │
┌───────┴─────────────────────┴───────────────────────────────┐
│                        CPU 0                                │
│                                                             │
│   1. 尝试获取 mutex                                         │
│   2. 成功：mutex 状态变为 LOCKED                            │
│   3. 现在可以安全访问 counter                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘

同时，CPU 1 尝试获取锁：

┌─────────────────────────────────────────────────────────────┐
│                        CPU 1                                │
│                                                             │
│   1. 尝试获取 mutex                                         │
│   2. 失败：mutex 已被 CPU 0 锁定                            │
│   3. 线程阻塞（等待）                                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 阶段 2：临界区内

```
┌─────────────────────────────────────────────────────────────┐
│                        主内存                                │
│                                                             │
│   counter = 0          mutex = LOCKED (by CPU 0)            │
│                                                             │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                        CPU 0                                │
│                                                             │
│   临界区内：                                                 │
│   1. LOAD  counter → EAX (得到 0)                           │
│   2. ADD   EAX, 1   → EAX = 1                               │
│   3. STORE EAX → counter                                    │
│                                                             │
│   此时 CPU 1 无法访问 counter                                │
│                                                             │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                        CPU 1                                │
│                                                             │
│   等待 mutex...                                             │
│   （线程被挂起，不消耗 CPU）                                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 阶段 3：释放锁（release）

```
┌─────────────────────────────────────────────────────────────┐
│                        CPU 0                                │
│                                                             │
│   unlock() 操作：                                           │
│   1. 将 mutex 改为 UNLOCKED                                 │
│   2. 这个操作带有 release 语义：                             │
│      - 确保临界区内的所有写入都提交到主内存                   │
│      - counter = 1 现在对其他 CPU 可见                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
                              │
                              │ 数据同步
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                        主内存                                │
│                                                             │
│   counter = 1  ← 已更新          mutex = UNLOCKED           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
                              │
                              │ 唤醒等待的线程
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                        CPU 1                                │
│                                                             │
│   1. 获取 mutex 成功                                        │
│   2. 读取 counter = 1（看到 CPU 0 的修改）                   │
│   3. 执行 counter++                                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### mutex 的关键语义

```
mutex.lock()    = acquire 语义
                  - 确保后续操作能看到之前的所有修改

mutex.unlock()  = release 语义  
                  - 确保之前的修改对其他线程可见
```

## 4. atomic 的作用

`atomic` 提供了更轻量的同步机制，直接在硬件层面保证操作的原子性。

### 基本用法

```cpp
#include <iostream>
#include <thread>
#include <atomic>

std::atomic<int> counter{0};  // 原子变量

void increment() {
    for (int i = 0; i < 100000; ++i) {
        counter++;  // 原子操作，不需要锁
    }
}

int main() {
    std::thread t1(increment);
    std::thread t2(increment);
    
    t1.join();
    t2.join();
    
    std::cout << "counter = " << counter << std::endl;  // 总是 200000
    return 0;
}
```

### atomic 的数据流动

```
┌─────────────────────────────────────────────────────────────┐
│                        主内存                                │
│                                                             │
│   atomic<int> counter = 0                                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
         ▲                                    ▲
         │                                    │
         │ 原子读-改-写                       │ 原子读-改-写
         │ (单条 CPU 指令)                    │ (单条 CPU 指令)
         │                                    │
┌────────┴────────┐                  ┌────────┴────────┐
│     CPU 0       │                  │     CPU 1       │
│                 │                  │                 │
│  LOCK XADD      │                  │  LOCK XADD      │
│  指令执行时：    │                  │  指令执行时：    │
│  - 锁住缓存行   │                  │  - 锁住缓存行   │
│  - 读取值       │                  │  - 读取值       │
│  - 加 1         │                  │  - 加 1         │
│  - 写回值       │                  │  - 写回值       │
│  - 释放缓存行   │                  │  - 释放缓存行   │
│                 │                  │                 │
└─────────────────┘                  └─────────────────┘
```

### 原子操作的硬件支持

```
普通操作（非原子）：
    LOAD → ADD → STORE  (三条独立指令)

原子操作：
    LOCK XADD  (一条不可分割的指令)

CPU 硬件保证：
    - 这条指令执行期间，其他 CPU 不能访问同一内存位置
    - 操作要么完全执行，要么完全不执行
    - 不会出现中间状态
```

### atomic vs mutex 的区别

```
┌─────────────────────────────────────────────────────────────┐
│                        mutex 方式                           │
│                                                             │
│   线程1: lock() → LOAD → ADD → STORE → unlock()            │
│                         │                                   │
│                         └─ 临界区（多条指令）                │
│                                                             │
│   线程2: lock() → [阻塞] → LOAD → ADD → STORE → unlock()   │
│                                                             │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                        atomic 方式                          │
│                                                             │
│   线程1: LOCK XADD (单条原子指令)                           │
│                                                             │
│   线程2: LOCK XADD (单条原子指令)                           │
│                                                             │
│   两条指令在硬件层面排队执行，不会交叉                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### atomic 的内存序

`atomic` 支持不同的内存序，控制操作的可见性：

```cpp
// 最强保证（默认）
counter.fetch_add(1, std::memory_order_seq_cst);

// 较弱保证（性能更好）
counter.fetch_add(1, std::memory_order_relaxed);
```

```
memory_order_relaxed:   只保证原子性，不保证顺序
memory_order_acquire:   后续操作能看到之前的修改
memory_order_release:   之前的修改对后续操作可见
memory_order_seq_cst:   最强保证，全局顺序一致
```

## 5. 对比总结

### 无锁 vs mutex vs atomic 的数据流动对比

```
┌─────────────────────────────────────────────────────────────┐
│                        无锁情况                              │
│                                                             │
│   CPU 0: LOAD → ADD → STORE                                 │
│   CPU 1: LOAD → ADD → STORE                                 │
│                                                             │
│   问题：两条指令序列可能交叉执行                              │
│   结果：数据竞争，结果不确定                                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                        mutex 情况                            │
│                                                             │
│   CPU 0: lock() → LOAD → ADD → STORE → unlock()            │
│   CPU 1: lock() → [等待] → LOAD → ADD → STORE → unlock()   │
│                                                             │
│   保证：临界区内的操作不会交叉                                │
│   开销：锁竞争、线程挂起/唤醒                                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                        atomic 情况                           │
│                                                             │
│   CPU 0: LOCK XADD                                          │
│   CPU 1: LOCK XADD                                          │
│                                                             │
│   保证：单条指令的原子性                                      │
│   开销：缓存行锁定，但无锁竞争                                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 何时选择哪种方案

| 场景 | 推荐方案 | 原因 |
|------|----------|------|
| **简单计数器** | `atomic` | 性能最好，代码简洁 |
| **复杂数据结构** | `mutex` | 保护多个相关变量 |
| **读多写少** | `shared_mutex` | 允许并发读取 |
| **高性能要求** | `atomic` + 内存序 | 避免锁开销 |
| **代码简单性** | `mutex` | 不易出错 |

### 选择指南

```
需要同步吗？
    │
    ├─ 是 → 操作简单？（单个变量，简单操作）
    │         │
    │         ├─ 是 → 使用 atomic
    │         │
    │         └─ 否 → 使用 mutex
    │
    └─ 否 → 不需要同步
```

### 实际建议

1. **默认使用 `mutex`**：简单、安全、不易出错
2. **性能关键时考虑 `atomic`**：但要理解内存序
3. **避免过度优化**：先保证正确性，再考虑性能
4. **使用 RAII 包装**：`std::lock_guard` 自动管理锁

```cpp
// 推荐：使用 lock_guard
{
    std::lock_guard<std::mutex> lock(mtx);
    counter++;
}  // 自动解锁

// 或使用 unique_lock（更灵活）
{
    std::unique_lock<std::mutex> lock(mtx);
    counter++;
}
```

## 总结

理解线程同步的关键在于理解数据在哪里流动：

1. **数据存在于多层存储**：寄存器、缓存、主内存
2. **没有锁时**：多个 CPU 可能同时修改同一数据，导致数据竞争
3. **mutex**：通过互斥锁保护临界区，确保操作的原子性
4. **atomic**：通过硬件指令保证单条操作的原子性
5. **选择依据**：根据操作复杂度、性能要求、代码复杂度综合考虑

记住：**锁保护的不是数据，而是操作数据的代码**。理解这一点，就能正确使用同步机制。

## 附录：数据位置变化详解

### A.1 数据存在哪里？

```
┌─────────────────────────────────────────────────────────────┐
│                        主内存 RAM                            │
│   所有 CPU 共享，速度最慢（~100ns）                           │
│   int counter = 0;  ← 数据的"真相"                           │
└─────────────────────────────────────────────────────────────┘
          │                    │
          ▼                    ▼
┌─────────────────────┐  ┌─────────────────────┐
│   CPU 0             │  │   CPU 1             │
│   ┌─────────────┐   │  │   ┌─────────────┐   │
│   │ L1 缓存     │   │  │   │ L1 缓存     │   │
│   │ L2 缓存     │   │  │   │ L2 缓存     │   │
│   └─────────────┘   │  │   └─────────────┘   │
│   ┌─────────────┐   │  │   ┌─────────────┐   │
│   │ 寄存器 EAX  │   │  │   │ 寄存器 EAX  │   │
│   │ 寄存器 EBX  │   │  │   │ 寄存器 EBX  │   │
│   └─────────────┘   │  │   └─────────────┘   │
└─────────────────────┘  └─────────────────────┘
```

**关键点**：
- 寄存器：每个 CPU 独立，其他 CPU 完全看不到
- 缓存：每个 CPU 有自己的 L1/L2，L3 共享
- 内存：所有 CPU 共享，速度最慢

### A.2 没有锁：数据位置变化

**代码**：
```cpp
int counter = 0;
counter++;
```

**反汇编**：
```asm
LOAD  counter → EAX    ; 从内存读到寄存器
ADD   EAX, 1           ; 寄存器加 1
STORE EAX → counter    ; 写回内存
```

**数据位置变化**：
```
步骤 1: LOAD
┌─────────────────────────────────────────────────────────────┐
│ 主内存: counter=0 ←──┐                                      │
└─────────────────────┼───────────────────────────────────────┘
                      │ 读取
┌─────────────────────┼───────────────────────────────────────┐
│ CPU 0: EAX=0        │ ← 数据在寄存器里                      │
└─────────────────────┴───────────────────────────────────────┘

步骤 2: ADD
┌─────────────────────────────────────────────────────────────┐
│ 主内存: counter=0  ← 没变！                                  │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│ CPU 0: EAX=1        ← 数据只在寄存器里                       │
└─────────────────────────────────────────────────────────────┘

步骤 3: STORE
┌─────────────────────────────────────────────────────────────┐
│ 主内存: counter=1  ← 现在写回                                │
└─────────────────────────────────────────────────────────────┘
```

**问题**：两个 CPU 同时执行，数据竞争！

```
CPU 0                           CPU 1
─────                           ─────
LOAD counter → EAX (得到 0)
EAX = 0+1 = 1
                                LOAD counter → EAX (得到 0)
                                EAX = 0+1 = 1
STORE EAX → counter (写入 1)
                                STORE EAX → counter (写入 1)
                                ← 覆盖了 CPU 0 的写入！

结果：counter = 1（期望 2）
```

### A.3 有 mutex：数据位置变化

**代码**：
```cpp
std::mutex mtx;
int counter = 0;
{
    std::lock_guard<std::mutex> lock(mtx);
    counter++;
}
```

**数据位置变化**：

```
阶段 1: 获取锁 (acquire)
─────────────────────────────────────────────────────────────────
CPU 0                              CPU 1
─────                              ─────
lock(mtx)
  │
  ├─ 原子指令（xchg）直接操作缓存行
  ├─ 绕过寄存器，直接修改缓存
  ├─ 缓存行状态: Modified
  │
  ├─ acquire 语义：
  │   ├─ 编译器不能把后续读写提前
  │   └─ 强制从缓存重新读取（不读寄存器缓存）
  │
  └─ 现在可以安全访问 counter

                                lock(mtx)
                                  │
                                  ├─ 失败（mutex 已被锁）
                                  └─ 线程阻塞，等待
```

```
阶段 2: 临界区内
─────────────────────────────────────────────────────────────────
CPU 0                              CPU 1
─────                              ─────
counter++
  │
  ├─ LOAD counter → EAX (从缓存读取，不是寄存器)
  ├─ EAX = 0+1 = 1
  └─ STORE EAX → counter (写入缓存)
     │
     ├─ L1 缓存: counter=1 (Modified)
     ├─ 其他 CPU 的副本失效
     └─ 数据在缓存里，不在内存！

                                等待...
                                （数据在 CPU 0 的缓存里）
                                （CPU 1 看不到）
```

```
阶段 3: 释放锁 (release)
─────────────────────────────────────────────────────────────────
CPU 0                              CPU 1
─────                              ─────
unlock(mtx)
  │
  ├─ release 语义：
  │   ├─ 编译器不能把之前的读写延迟
  │   └─ 强制把缓存写入刷新到共享缓存
  │
  └─ 缓存行状态: Modified → Shared/Invalidated

                                  lock(mtx) 成功
                                    │
                                    ├─ acquire 语义：
                                    │   └─ 强制从缓存重新读取
                                    │
                                    └─ counter = 1（看到 CPU 0 的修改）
```

### A.4 有 atomic：数据位置变化

**代码**：
```cpp
std::atomic<int> counter{0};
counter++;
```

**反汇编**：
```asm
LOCK XADD [counter], EAX  ; 原子指令：读-改-写
```

**数据位置变化**：

```
步骤 1: 原子读-改-写
─────────────────────────────────────────────────────────────────
CPU 0                              CPU 1
─────                              ─────
LOCK XADD
  │
  ├─ 硬件保证：
  │   ├─ 独占缓存行（MESI 协议）
  │   ├─ 读取旧值
  │   ├─ 计算新值
  │   ├─ 写回新值
  │   └─ 释放缓存行
  │
  ├─ 全程在缓存里完成，不经过通用寄存器
  └─ 其他 CPU 不能同时修改同一缓存行

                                  等待缓存行...
                                  │
                                  └─ 缓存行释放后，获取新值
```

**与 mutex 的区别**：
```
mutex:     lock() → LOAD → ADD → STORE → unlock()
              │                              │
              └──────── 临界区 ──────────────┘
              （多条指令，需要同步）

atomic:    LOCK XADD
              │
              └─ 单条原子指令
              （硬件保证不可分割）
```

### A.5 总结：数据位置变化对比

| 阶段 | 无锁 | mutex | atomic |
|------|------|-------|--------|
| **初始** | 内存 | 内存 | 内存 |
| **读取** | 寄存器 | 缓存（acquire 后） | 缓存（原子指令） |
| **计算** | 寄存器 | 寄存器 | 缓存（原子指令） |
| **写入** | 内存（可能覆盖） | 缓存（release 后刷新） | 缓存（原子指令） |
| **可见性** | 无保证 | acquire/release 保证 | 硬件保证 |

**一句话**：锁和原子操作都是通过硬件层面的缓存一致性和内存屏障来保证数据可见性的。
